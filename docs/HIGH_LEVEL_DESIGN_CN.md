## 1. 概述

**MyAgent** 是一个基于 Python 构建的自托管个人办公 AI 助手。它将 **ReAct（推理-行动）智能体循环**与 **OpenClaw 兼容技能发现**、**多模型路由**、**四层记忆系统**和**多通道消息**（CLI、飞书、Telegram）整合为一个可独立部署的完整单元。

系统专为**单用户深度使用**设计——而非多租户 SaaS——这简化了认证、数据隔离和扩展需求，同时通过长期记忆和技能自学习实现深度个性化。

### 关键指标

| 指标 | 数值 |
|---|---|
| 内置工具 | 12 个 |
| 可发现技能 | 200+（OpenClaw 生态） |
| 记忆层级 | 4 层（工作 → 情景 → 语义 → 用户画像） |
| LLM 提供商 | 5 个（智谱、DeepSeek、Google、OpenAI、Ollama/本地） |
| 消息通道 | 3 个（CLI、飞书、Telegram） |
| 通道看门狗 | 自动重启 + 指数退避 |
| ReAct 最大迭代 | 60 轮 |
| 默认端口 | 5196 |

## 4. 核心子系统

### 4.1 智能体编排器（ReAct 循环）

编排器是 MyAgent 的核心。它实现了 **ReAct（Reason + Act）** 模式——一个迭代循环，LLM 推理问题、选择并调用工具、观察结果，然后继续直到任务完成。

**执行流程：**

```
用户消息
     │
     ▼
┌─────────────────────────┐
│ 构建系统提示              │  ← 用户上下文 + 技能目录 + 记忆
│ 加载工具 Schema           │
└────────────┬────────────┘
             │
             ▼
      ┌─────────────┐
      │  LLM 调用    │◄──────────────────────┐
      └──────┬──────┘                         │
             │                                │
      ┌──────▼──────┐                         │
      │ 有工具调用?   │                        │
      └──┬───────┬──┘                         │
         │       │                            │
      否 │       │ 是                          │
         │       ▼                            │
         │  ┌────────────┐                    │
         │  │ 执行        │                   │
         │  │ 工具调用     │                   │
         │  └─────┬──────┘                    │
         │        │                           │
         │        ▼                           │
         │  ┌────────────┐                    │
         │  │ 追加到      │                   │
         │  │ 消息列表    │───────────────────┘
         │  └────────────┘
         │
         ▼
  ┌──────────────┐
  │ 存储到记忆    │
  │ 触发学习评估  │
  └──────────────┘
```

**关键行为：**

| 方面 | 细节 |
|---|---|
| **最大迭代** | 60 轮（通过 `max_iterations` 可配置） |
| **全局超时** | `max_iterations × default_timeout` 秒 |
| **迭代耗尽处理** | 请求 LLM 生成当前进度的中间总结，而非静默失败 |
| **对话历史** | 注入最近 20 条消息作为上下文 |
| **记忆集成** | 每次对话前检索上下文；完成后存储交互 |
| **学习触发** | 任务完成后，`LearningLoop.on_task_complete()` 评估复杂度 |

**系统提示组装：**

```
┌─────────────────────────────────────┐
│           系统提示                    │
│                                     │
│  1. 角色与规则（静态）                │
│  2. 用户上下文（来自 L4 记忆）        │
│  3. 技能目录（仅摘要）                │
│  4. 记忆上下文（情景回忆）            │
│                                     │
└─────────────────────────────────────┘
```

### 4.2 LLM 路由与降级链

#### LLM 路由器

路由器将 `provider/model` 引用映射到具体的 LLM 提供商客户端。它支持**任务类型路由**——不同任务类别发送到不同模型。

**路由解析顺序：**

```
1. 是否有显式 model 参数?       → 使用该模型
2. 任务类型是否在路由表中?      → 使用路由的模型
3. 使用默认模型                 → 兜底
```

**模型引用格式：**

所有模型引用使用 `提供商/模型` 约定：

```
zai/glm-5.1              →  智谱 AI → GLM-5.1
deepseek/deepseek-coder  →  DeepSeek → Coder
ollama/qwen3.6:35b       →  本地 Ollama → Qwen 3.6 (35B)
google/gemini-2.5-pro    →  Google → Gemini 2.5 Pro
```

**提供商初始化：**

```
┌──────────────────────────────────────────┐
│              提供商注册表                  │
│                                          │
│  ┌─────────┐  ┌──────────┐  ┌────────┐  │
│  │  智谱    │  │ DeepSeek │  │ Google │  │
│  │  (ZAI)  │  │          │  │        │  │
│  └─────────┘  └──────────┘  └────────┘  │
│  ┌─────────┐  ┌──────────────────────┐  │
│  │ OpenAI  │  │ Ollama (始终初始化)   │  │
│  │         │  │ localhost:11434      │  │
│  └─────────┘  └──────────────────────┘  │
└──────────────────────────────────────────┘
```

> Ollama 始终作为本地兜底初始化，即使未在配置中显式指定。

#### 降级链

`FallbackChain` 在主模型失败时提供**自动降级**。它实现了**三振规则**：连续失败 3 次后，模型临时移出轮换队列。

```
请求 → zai/glm-5.1 ──── 失败 ──► openrouter/auto ──── 失败 ──► ollama/qwen3.5:4b
            │                         │                          │
            └── 失败计数 += 1         └── 失败计数 += 1           └── 最后兜底
            └── 计数 < 3: 重试        └── 计数 < 3: 重试
            └── 计数 ≥ 3: 跳过        └── 计数 ≥ 3: 跳过
```

**线程安全：** 使用 `asyncio.Lock` 懒初始化，由 `threading.Lock` 保护，防止多线程上下文中的竞态条件。

### 4.3 工具注册中心

所有工具实现统一的 `BaseTool` 接口，以 OpenAI function-calling 格式返回结果。

**工具接口定义：**

```python
class BaseTool(ABC):
    @abstractmethod
    def get_schema(self) -> dict:
        """返回 OpenAI function-calling 格式的 Schema"""

    @abstractmethod
    async def execute(self, **kwargs) -> ToolResult:
        """使用给定参数执行工具"""

@dataclass
class ToolResult:
    success: bool
    output: str
    error: str = ""
```

**内置工具一览：**

| 工具 | 输入 | 输出 | 安全说明 |
|---|---|---|---|
| `exec` | Shell 命令字符串 | stdout/stderr | 受危险命令模式拦截 |
| `file_read` | 文件路径 | 文件内容 | — |
| `file_write` | 路径 + 内容 | 写入确认 | — |
| `file_edit` | 路径 + 旧/新文本 | 编辑确认 | — |
| `web_search` | 查询字符串 | 搜索结果 | 需要 API Key |
| `web_fetch` | URL | 提取的文本 | — |
| `email` | 收件人/主题/正文 | 发送确认 | 需要应用专用密码 |
| `calendar` | 日期/查询 | 事件列表 | — |
| `data_analysis` | 数据 + 问题 | 分析结果 | — |
| `deep_search` | 主题 | 多源报告 | 聚合多次搜索 |
| `feishu_api` | 文档/消息 ID | 内容/消息 | 需要应用凭据 |

### 4.4 技能系统

技能系统通过 OpenClaw 兼容的 `SKILL.md` 文件提供**动态扩展**能力。技能在启动时发现，以摘要形式注入系统提示，在需要时按需加载完整内容。

#### 4.4.1 技能发现

技能从多个目录根按**反优先级顺序**（最低优先级先扫描）扫描，高优先级目录的同名技能覆盖低优先级：

```
优先级（高 → 低）：
  ~/.openclaw/workspace/skills                     ← 用户自定义（覆盖所有）
  ~/.openclaw/workspace/.agents/skills             ← 智能体专属
  ~/.agents/skills                                 ← 通用技能
  ~/.openclaw/skills                               ← 已安装技能
  ~/.npm-global/lib/node_modules/openclaw/skills   ← 内置技能（基础）
```

**资格检查：**

```
对于每个发现的 SKILL.md：
  ├── 平台过滤（os: macos, linux, windows）
  ├── 二进制依赖（requires.bins → shutil.which）
  ├── 任一二进制（requires.anyBins → 至少一个存在）
  └── Python 包（requires.pip → importlib.util.find_spec）
```

#### 4.4.2 技能解析器

解析 OpenClaw `SKILL.md` 格式，包含 YAML 前置元数据：

```yaml
---
name: my-skill
description: "技能功能描述"
metadata:
  openclaw:
    emoji: "🔧"
    os: [linux, macos]
    requires:
      bins: [ffmpeg, curl]
      pip: [pandas>=2.0]
---

# 技能使用说明

智能体按步骤执行的指令...
```

**解析结果：**

```
OpenClawSkill
  ├── meta: SkillMeta（名称、描述、emoji、平台过滤、依赖...）
  ├── instructions: str（Markdown 正文）
  ├── skill_dir: Path（包含 SKILL.md 的目录）
  └── source: str（workspace | managed | personal | bundled）
```

#### 4.4.3 技能执行器

**两阶段加载**策略，最小化 Token 消耗：

| 阶段 | 触发时机 | 注入内容 |
|---|---|---|
| **目录** | 启动时 → 系统提示 | 名称 + emoji + 一行描述（最多 50 个技能） |
| **完整加载** | 按需（LLM 匹配到技能时） | 完整 Markdown 指令注入到对话中 |

#### 4.4.4 技能学习器（自学习）

学习循环自动从复杂任务模式中提取新技能：

```
任务完成
     │
     ▼
评估复杂度 (0-1)
  ├── 工具调用数量（≥5: +0.3）
  ├── 工具多样性（≥3 种: +0.2）
  ├── 结果长度（>1000 字符: +0.2）
  └── 关键词（分析/报告/研究: +0.1）
     │
     ▼
复杂度 > 0.6 且 工具调用 ≥ 3?
     │
  是 │               否 → 仅记录日志
     ▼
LLM 提取 SKILL.md
     │
     ▼
保存到 _pending/（等待人工审核）
```

待审核的技能可以批准（移至正式技能目录）或拒绝（删除）。

### 4.5 四层记忆系统

受人类认知启发，记忆系统分为四层，每层具有不同的存储、检索和保留特性：

```
┌────────────────────────────────────────────────────────────┐
│                       记忆管理器                             │
│                                                             │
│  L1：工作记忆                  L2：情景记忆                  │
│  ┌───────────────────┐      ┌───────────────────────┐      │
│  │ 存储：内存          │      │ 存储：SQLite + FTS5    │      │
│  │ 内容：消息列表      │      │ 内容：交互摘要         │      │
│  │ 保留：会话期        │      │ 保留：7 天             │      │
│  │ 容量：最近 20 条    │      │ 搜索：全文检索         │      │
│  │ 访问：FIFO         │      │ 访问：FTS5 + 时间排序  │      │
│  └───────────────────┘      └───────────────────────┘      │
│                                                             │
│  L3：语义记忆                  L4：用户画像                  │
│  ┌───────────────────┐      ┌───────────────────────┐      │
│  │ 存储：sqlite-vec   │      │ 存储：JSON 文件        │      │
│  │ 内容：向量嵌入      │      │ 内容：用户偏好档案     │      │
│  │ 保留：永久          │      │ 保留：永久             │      │
│  │ 搜索：余弦相似度    │      │ 访问：键值对           │      │
│  │ 嵌入模型：bge-m3   │      │ 行为模式、使用偏好     │      │
│  └───────────────────┘      └───────────────────────┘      │
│                                                             │
└────────────────────────────────────────────────────────────┘
```

**记忆操作：**

| 操作 | 涉及层级 | 流程 |
|---|---|---|
| **存储** | L1、L2、L4 | 工作记忆 ← 消息；情景记忆 ← 摘要 + 重要度；用户画像 ← 行为模式 |
| **回忆** | L2、L3 | 情景记忆 FTS5 搜索 + 最近条目；语义记忆向量相似度 |
| **上下文构建** | 全部 | 用户画像 + 最近情景 + 相关记忆 → 注入系统提示 |
| **记忆巩固** | L2 → L3 | 高重要度的情景条目转化为向量嵌入长期保存 |

**重要度评分公式（L2）：**

```
基础分 = 0.3
+ 0.2  如有工具调用
+ 0.1  如响应 > 500 字符
+ 0.15 如包含关键词（记住、重要、决定、分析、报告...）
= min(总分, 1.0)
```

**保留策略：**

- **L1 工作记忆**：会话级，重启后清空
- **L2 情景记忆**：默认 7 天；重要度 ≤ 0.7 的条目定期清理；重要度 > 0.7 永久保留
- **L3 语义记忆**：永久保留（向量嵌入）
- **L4 用户画像**：永久保留（JSON 文件）

### 4.6 多智能体编排

对于复杂任务，MyAgent 可以生成**子智能体**并行执行：

```
复杂任务
     │
     ▼
任务分解（LLM）
     │
     ├── "子任务 1"
     ├── "子任务 2"
     └── "子任务 3"
          │
          ▼
  ┌────────────────────────────────┐
  │      asyncio.gather()          │
  │                                │
  │  子智能体1   子智能体2   子智能体3    ← 独立实例
  │  (5 轮)     (5 轮)     (5 轮)       ← 有限迭代
  │    │          │          │
  └────┼──────────┼──────────┘
       │          │          │
       ▼          ▼          ▼
  结果聚合（LLM 汇总）
       │
       ▼
  连贯的最终回复
```

**关键设计决策：**
- 每个 `SubAgent` 是**全新实例**——子智能体之间无共享状态
- 子智能体限制为 **5 轮迭代**（主智能体为 60 轮）
- 结果由专用 LLM 调用汇总合成
- 如分解失败，降级为直接执行原始任务

### 4.7 网关与通道集成

网关是一个 **FastAPI** 应用，充当中央消息分发器。通道作为**独立子进程**运行，通过内部 HTTP 端点与网关通信。

```
┌─────────────────────────────────────────────────────────┐
│                    网关 (FastAPI)                         │
│                                                          │
│  端点：                                                   │
│  ├── GET  /                → 监控面板 (SPA HTML)          │
│  ├── WS   /ws              → WebSocket 实时对话           │
│  ├── GET  /api/status      → 智能体状态（仅本地）          │
│  ├── GET  /api/scheduler   → 定时任务状态                 │
│  ├── POST /webhook/feishu       → 飞书 Webhook           │
│  ├── POST /internal/feishu_message   → HMAC 保护          │
│  └── POST /internal/telegram_message → HMAC 保护          │
│                                                          │
│  安全：                                                   │
│  ├── CORS（仅限 localhost）                               │
│  ├── 内部端点 HMAC 签名验证                                │
│  └── 本地绑定 (127.0.0.1)                                │
└─────────────────────────────────────────────────────────┘
         ▲                              ▲
         │ HTTP（HMAC 签名）             │ HTTP（HMAC 签名）
    ┌────┴─────┐                  ┌─────┴──────┐
    │  飞书     │                  │  Telegram   │
    │  子进程   │                  │  子进程      │
    │          │                  │             │
    │lark-oapi │                  │  aiogram    │
    │WebSocket │                  │  polling    │
    └──────────┘                  └─────────────┘
```

**为何使用子进程？**

- 通道 SDK（lark-oapi、aiogram）有各自的事件循环和生命周期
- 子进程隔离防止通道崩溃影响整个网关
- 每个子进程将消息 POST 到网关的内部端点，由 `AgentOrchestrator` 分发处理

**初始化保护：**

`AgentOrchestrator` 在首次请求时**延迟初始化**，由 `asyncio.Lock` + double-check 模式保护。初始化过程在 `asyncio.to_thread()` 中运行，避免阻塞事件循环（SQLite 建库、技能扫描等同步操作）。

### 4.8 通道看门狗

通道看门狗是一个**后台监控守护任务**，定期检测所有注册通道子进程的存活状态，发现崩溃时自动重启。它填补了一个关键运维缺口：当通道子进程因 DNS 解析失败、网络中断或 SDK 错误而崩溃时，网关此前无法检测或恢复故障。

**架构：**

```
┌─────────────────────────────────────────────────────────────┐
│                    ChannelWatchdog                            │
│                                                              │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  后台 asyncio.Task（每 30 分钟执行一次）               │   │
│  │                                                       │   │
│  │  遍历每个注册的通道：                                   │   │
│  │    ├── is_alive()?                                    │   │
│  │    │   ├── 是 → 重置失败计数                           │   │
│  │    │   └── 否 → 检查退避窗口                           │   │
│  │    │       ├── 退避中 → 跳过                          │   │
│  │    │       ├── 超过最大重试 → 进入冷却                  │   │
│  │    │       └── 可重试 → stop() → start() → 重启       │   │
│  │    └── 更新 next_retry_after（指数退避）                │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  已注册的通道：                                                │
│  ├── 飞书：lark-oapi WebSocket 子进程                        │
│  └── Telegram：aiogram polling 子进程                        │
│                                                              │
│  指数退避序列：                                                │
│  第1次：60秒 | 第2次：120秒 | 第3次：240秒                    │
│  第4次：480秒 | 第5次：960秒 → 冷却（下个 30 分钟周期）       │
└─────────────────────────────────────────────────────────────┘
```

**关键参数：**

| 参数 | 默认值 | 说明 |
|---|---|---|
| `check_interval` | 1800秒（30分钟） | 通道健康检测间隔 |
| `max_retries` | 5 | 单通道最大连续重试次数，超过后进入冷却 |
| `backoff_base` | 60秒 | 指数退避基数 |
| `backoff_max` | 3600秒（1小时） | 退避上限 |

**退避序列：**

```
第1次失败 → 等待 60秒   → 重启
第2次失败 → 等待 120秒  → 重启
第3次失败 → 等待 240秒  → 重启
第4次失败 → 等待 480秒  → 重启
第5次失败 → 等待 960秒  → 重启
第6次失败 → 冷却（等下个 30 分钟周期，重置计数器）
```

**状态报告：**

看门狗通过 `/api/status` 暴露运行状态：

```json
{
  "watchdog": {
    "enabled": true,
    "check_interval": 1800,
    "channels": {
      "feishu": { "alive": true, "consecutive_failures": 0, "total_restarts": 1 },
      "telegram": { "alive": false, "consecutive_failures": 3, "total_restarts": 5, "next_retry_in": 240 }
    }
  }
}
```

**注册机制：**

通道在 `_init_agent()` 中通过 `watchdog.register(name, start_fn, stop_fn, is_alive_fn)` 注册。看门狗在 `Gateway.start()` 中所有通道启动后启动。

**配置（`config/default.yaml`）：**

```yaml
channels:
  watchdog:
    enabled: true
    check_interval: 1800   # 30 分钟
    max_retries: 5
    backoff_base: 60
    backoff_max: 3600
```

### 4.9 定时任务调度器

基于 **APScheduler**（AsyncIO 模式），支持标准 cron 表达式。

```
┌──────────────────────────────────────┐
│          CronScheduler               │
│                                      │
│  register_job(name, cron, handler)   │
│         │                            │
│         ▼                            │
│  CronTrigger.from_crontab(cron)      │

│         │                            │

│    │  handler │                        │

│         ▼                            │
```

- 日志写入使用 `asyncio.to_thread()` 避免阻塞事件循环

## 8. 安全架构

### 威胁模型

| 威胁 | 缓解措施 |
|---|---|
| **网络暴露** | 默认绑定 `127.0.0.1`；CORS 限制为 localhost |
| **内部端点滥用** | 所有 `/internal/*` 端点要求 HMAC-SHA256 签名 |
| **Shell 注入** | 17 种危险命令模式正则拦截 |
| **PII 泄露** | 6 种 PII 模式检测器，审计日志自动脱敏 |
| **凭据暴露** | `.env` 加入 `.gitignore`；密钥不存入 YAML |
| **子进程入侵** | 通道在隔离的子进程中运行 |
| **模型故障** | 三振降级链跨提供商切换 |

### 纵深防御

```
第 1 层：网络       → 本地绑定、CORS 白名单
第 2 层：传输       → 内部通信的 HMAC 签名
第 3 层：应用       → PII 检测、命令拦截
第 4 层：数据       → 审计日志自动脱敏
第 5 层：进程       → 通道子进程隔离
第 6 层：韧性       → 通道看门狗自动重启
第 7 层：模型       → 降级链保障韧性
```

## 10. 设计决策与权衡

| 决策 | 理由 | 权衡 |
|---|---|---|
| **单用户设计** | 消除多租户复杂度；实现深度个性化 | 无法同时服务多用户 |
| **子进程通道** | 隔离 SDK 崩溃；清晰的生命周期管理 | 有一定开销；需要 HMAC 内部认证 |
| **两阶段技能加载** | 保持系统提示精简（~50 个技能摘要） | 完整指令在对话中加载，增加延迟 |
| **SQLite 存储** | 零配置；文件级；内置 FTS5 | 不适合高并发或分布式部署 |
| **ReAct 循环替代先规划后执行** | 更灵活；自然处理意外的工具结果 | 简单任务消耗较多 Token；无全局计划可见性 |
| **三振降级** | 简单有效；提供商恢复后自动重置 | 对瞬时故障可能过于激进 |
| **通道看门狗（30分钟）** | 自愈无需人工干预；指数退避防止疯狂重连 | 30分钟轮询意味着最长 30 分钟检测延迟 |
| **延迟初始化智能体** | 网关快速启动；首次请求无阻塞 | 首次请求有冷启动延迟（技能扫描、DB 初始化） |
| **YAML + .env 配置** | 熟悉的格式；密钥与行为分离 | 需管理两个文件；不支持热加载 |
| **全面 asyncio** | HTTP、WebSocket、LLM 调用原生异步 | 需谨慎的锁管理；混合同步/异步代码 |
| **技能自学习** | 随时间减少手动技能创建 | 学习的技能需人工审核；质量参差不齐 |
